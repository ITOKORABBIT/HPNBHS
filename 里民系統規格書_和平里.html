1. 專案目標

  • 提供里民可以用手機直接送出案件通報、查詢公開案件、查看活動公告與特約商店資訊。
  • 提供里長端可集中管理案件、審核商店、發佈公告與維護管理者帳號。
  • 以免安裝 App 的方式建置，透過 GitHub Pages 提供公開頁面，降低使用門檻。
  • 透過 Google Apps Script 串接 Google Sheets 與 Google Drive，降低建置與後續維護成本。
  • 透過 Make Webhook 將新案件、新商店申請等事件推播到既有通知流程。
免安裝 手機優先 Google 生態整合 里長後台管理

2. 角色與權限

角色 登入方式 可操作內容
里民端 免登入瀏覽公開頁，送出案件或商店申請時僅填表單資料 送出案件通報、送出特約商店申請、查閱公開案件清單、查閱商店清單、閱讀公告
管理者 Google Sign-In 驗證 + 管理員名單授權 查看後台摘要、進入案件管理、審核商店、管理公告
超級管理者 Google Sign-In 驗證 + 管理員名單角色判斷 除一般管理外，可新增、編輯、停用、刪除管理者帳號

3. 功能模組總覽

模組 主要頁面 現行功能
里民案件通報 `report.html`、`openlist.html`、`opendetail.html`、`list.html`、`detail.html` 里民可送出案件、上傳照片、填寫地點與 1999 案號；里長端可更新狀態、填寫回覆、設定公開內容與上傳處理照片。
特約商店申請 `store.html`、`storeopenlist.html`、`storeopendetail.html`、`storelist.html`、`storedetail.html` 店家可送出申請；里長端可審核狀態、整理公開內容、設定方案等級與置頂排序。

管理後台 `admin.html` 顯示案件、商店、公告、管理員四大頁籤摘要；可快速進入對應管理頁或公開頁。

5. 主要流程說明

5-1. 里民案件流程

  • 里民於 `report.html` 填寫姓名、電話、類別、主旨、描述、地址、地圖與照片後送出。
  • 前端將資料送至 Apps Script `submitReport`，寫入 Google Sheet「案件清單」工作表。
  • 系統建立案件編號、回覆網址、照片連結，並透過 Make Webhook 送出新通報通知。
  • 里長在 `list.html` 與 `detail.html` 進行回覆、更新狀態、決定是否公開。
  • 勾選公開後，資料會出現在 `openlist.html` 與 `opendetail.html`。

5-2. 特約商店流程

  • 店家於 `store.html` 送出申請，含聯絡資料、店家資訊、優惠方案、營業時間與照片。
  • Apps Script `submitStore` 進行基本驗證、防機器人與節流處理後寫入 Google Sheet。
  • 里長在 `storelist.html` / `storedetail.html` 審核，決定是否公開、調整公開欄位、方案等級與置頂排序。
  • 狀態為公開後，資料會出現在 `storeopenlist.html` 與 `storeopendetail.html`。

5-3. 公告流程

  • 管理者於 `bulletinlist.html` 建立或編輯公告，可設定標題、內容、圖片、草稿或發佈、是否置頂。
  • Apps Script `addBulletin` / `updateBulletin` 寫入 Google Sheet「公告」工作表。
  • `bulletin.html` 僅讀取已發佈的公告，並依置頂與時間排序顯示。

5-4. 管理者流程

  • `admin.html` 透過 Google Sign-In 登入後讀取四大模組摘要。
  • 僅具超級管理權限者可對 `getAdmins`、`addAdmin`、`updateAdmin`、`deleteAdmin` 進行操作。

6. 資料結構

資料表 主要欄位 用途
案件清單 案件編號、狀態、類別、聯絡人、地址、照片、回覆內容、公開欄位、置頂順序 儲存里民通報原始資料、里長回覆、公開摘要與照片。
商店清單 商店編號、申請人資料、店家資料、公開欄位、方案等級、置頂順序 儲存特約商店申請、審核與公開內容。
管理員 顯示名稱、角色、啟用狀態、Email 作為 Google 登入授權名單與後台權限來源。
公告 公告編號、建立時間、標題、內容、圖片、置頂、狀態、作者 儲存公告欄內容與發佈狀態。

7. 技術架構

層級 技術 說明
前端網站 HTML / CSS / Vanilla JavaScript / GitHub Pages 所有使用者頁面皆為靜態頁面，直接由 GitHub Pages 提供。
資料存取 API Google Apps Script Web App 負責登入驗證、資料讀寫、圖片上傳、權限檢查與資料轉換。
資料庫 Google Sheets 以工作表方式儲存案件、商店、管理員與公告資料。
圖片儲存 Google Drive 里民與管理端上傳圖片後，轉為公開檢視連結供前端載入。
登入 Google Identity Services 管理端使用 Google 帳號登入，再由後端比對授權名單。
通知推播 Make Webhook 新案件與新商店申請可推送到既有通知情境。

8. 非功能需求

  • 介面需支援手機瀏覽，主要操作以觸控為主。
  • 管理端需具備登入與角色判斷，避免未授權帳號讀寫資料。
  • 公開上傳需有 API Key、來源檢查、節流限制與檔案大小限制。
  • 公告、案件、商店等模組需在無額外資料庫情況下可維持基本維運。
  • 頁面視覺需延續目前系統的米白、綠色、金色系設計語言。
備註：目前系統屬於輕量型里政資訊平台，部署快速、維護成本低，但也需持續留意 Google Apps Script 配額、Google Drive 權限與 GitHub Pages 靜態部署限制。
